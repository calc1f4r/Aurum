# 2) PDAs & seeds (deterministic addresses you must enforce)

Use program id `P`:

* Global config account (singleton)

  * Seed: `["global", program_id]` (or just `["global"]`)
  * PDA derived: `Pubkey::find_program_address(&[b"global"], &P)`

* Market / Reserve (one per asset mint)

  * Seed: `["reserve", mint_pubkey]`
  * Holds reserve metadata and borrows/collateral mints.

* Reserve vault (liquidity token account owned by program)

  * Seed: `["vault", reserve_pubkey]` (or `["vault", mint_pubkey]`)
  * Token account owned by token program with owner = PDA.

* Reserve collateral mint (if you issue cTokens like representation)

  * Seed: `["collateral_mint", reserve_pubkey]` (optional; or use mint created off-chain)

* Obligation / UserPosition (per user per market or single aggregated)

  * Seed: `["obligation", user_pubkey, reserve_pubkey]` — *isolated per user-per-reserve* (smaller and simpler) OR
  * Seed: `["obligation", user_pubkey]` — allowing multiple positions inside account (more complex gas)

* Fee receiver account

  * Seed: `["fees", reserve_pubkey]` — SPL token account PDA.

* Interest model config (could be inside Reserve account)

Rationale: PDAs prevent phantom accounts and unauthorized owners. Use explicit seeds with canonical ordering.

```rust
use anchor_lang::prelude::*;
use anchor_spl::token::TokenAccount;

pub const RAY: u128 = 1_000_000_000_000_000_000_000_000_000u128; // 1e27

#[account]
pub struct GlobalConfig {
    pub admin: Pubkey,                // 32
    pub paused: bool,                 // 1
    pub treasury: Pubkey,             // 32
    pub allowed_mints: Vec<Pubkey>,   // 4 + 32 * n   (dynamic)
    pub pyth_program: Pubkey,         // 32
    pub fee_bps: u16,                 // 2 (basis points)
    pub bump: u8,                     // 1
    // packing/padding
}

#[account]
pub struct Reserve {
    pub mint: Pubkey,                 // underlying asset mint (32)
    pub vault: Pubkey,                // token account PDA holding liquidity (32)
    pub collateral_mint: Pubkey,      // if issued (32)
    pub total_borrows: u128,          // in underlying scaled units (16)
    pub total_supply: u128,           // total collateral token supply (16)
    pub last_updated_ts: i64,         // i64 unix (8)
    pub interest_rate_model: InterestRateModel, // ~56-72 bytes
    pub reserve_factor: u16,          // 2 (portion to treasury, in bps)
    pub liquidation_threshold: u16,   // 2 (bps)
    pub ltv: u16,                     // 2 (bps)
    pub fee_receiver: Pubkey,         // 32
    pub bump: u8,                     // 1
}
 // estimated fixed size: ~ 32*5 + 16*2 + 8 + ~80 + small = ~300 bytes

#[derive(AnchorSerialize, AnchorDeserialize, Clone)]
pub struct InterestRateModel {
    pub base_rate: u128,      // RAY scaled
    pub slope1: u128,         // RAY
    pub slope2: u128,         // RAY
    pub kink: u128,           // utilization at kink as RAY (0..RAY)
}

#[account]
pub struct Obligation {
    pub owner: Pubkey,           // 32
    pub reserve: Pubkey,         // 32 (the asset)
    pub deposited: u128,         // collateral amount (raw)
    pub borrowed: u128,          // borrow amount (raw + interest index applied)
    pub borrow_index: u128,      // last interest index (RAY)
    pub last_updated_ts: i64,
    pub bump: u8,
}
 // size ~ 32+32+16+16+16+8+1 = 121 bytes
```

Notes:

* Use `u128` for monetary values and RAY (1e27) for scaled fractional math to avoid precision loss across multiply/divide.
* For token amounts stored as u64 on chain, convert to RAY-scaled values when computing interest. Keep raw `u64` in vault; use `u128` for accumulators.

# 4) Interest maths (formal)

Let:

* $C$ = cash in vault (raw underlying, u128)
* $B$ = total borrows outstanding (u128)
* $U = \frac{B}{C + B}$ utilization (0..1), compute in RAY
* Interest model piecewise:

  * If $U \le k$: $r = r_{base} + slope1 * \frac{U}{k}$
  * Else: $r = r_{base} + slope1 + slope2 * \frac{U - k}{1-k}$
* Rates are expressed as annualized rate in RAY (e.g., 0.05 RAY = 5%/year).
* Convert annual rate to per-second factor:

  * $r_{sec} = \frac{r}{31{,}557{,}600}$ (seconds/year; use 31,557,600)
* Accrual over $\Delta t$ seconds:

  * interest factor: $f = 1 + r_{sec} \cdot \Delta t$ (approx, small delta). For higher precision, use exponent: $f = e^{r_{sec} \Delta t}$ but linear is fine for short intervals onchain.
* Borrow index update:

  * `borrow_index_new = borrow_index_old * f` (in RAY arithmetic)
* New borrows: `total_borrows *= f;` and interest to reserves = `(total_interest * reserve_factor)`

Implementation notes:

* Use fixed point RAY arithmetic: `mul_ray(a,b) = (a*b)/RAY`.
* For safety use checked arithmetic, saturating if necessary, and revert on overflow.

# 5) Health factor, LTV, liquidation

Define `price_pyth` from Pyth for each mint. Use oracle checks (below).

Collateral value:

$$
V_c = deposit_{raw} \times price \times collateral\_factor
$$

Borrowed value:

$$
V_b = borrowed_{raw} \times price_{borrowed}
$$

Health factor:

$$
HF = \frac{V_c}{V_b}
$$

Liquidation condition:

$$
HF < 1.0 \quad\text{(or } HF < liquidation\_threshold \text{ if defined)}
$$

Liquidator may repay up to `max_liquidation_close = min(borrowed * close_factor, )` and seize collateral discounted by `discount = 1 + incentive_bps`.

Quantities must be computed in RAY with careful rounding (round down for collateral taken, round up for debt repaid to avoid under-collateralization).

# 10) Security checklist (must implement)

* Oracle safety:

  * Price staleness checks (configurable)
  * Reject implausible large price jumps beyond threshold
  * Optionally require TWAP or second oracle for high value markets
* CPI safety:

  * Check token accounts ownership, `mint` matches expected, `owner` matches signer (for user)
* Reentrancy:

  * Solana’s model is single-threaded per tx but attacks via cross-program calls possible. Use account constraints order: read-only accounts first; write accounts locked. Use `require!( !paused )`.
* Authorization:

  * Admin-only operations have signer verification.
* Arithmetic:

  * Use checked\_mul/checked\_div with saturating guards. Prevent overflow on `u128` by bounding.
* Liquidation:

  * Limit `close_factor` and `liquidation_incentive` to sane values (e.g., close\_factor <= 0.5, incentive <= 0.10).
* Flashloan risk:

  * Because you allow same-tx operations, ensure that liquidation & borrow checks are snapshot-consistent; always compute health after accrual and before transfer.
* Closing accounts:

  * Return lamports to a safe recipient; ensure no fund loss.
* CPI replays:

  * Check all account owners; disallow arbitrary PDAs being passed without PDA verification.
